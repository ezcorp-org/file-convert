  <files>apps/frontend/tests/TEST_AUDIT.md</files>
  <action>
Read every test file in apps/frontend/tests/ (26 files total). For each file, evaluate:

1. **Coverage value**: Does it test a critical user journey? Does it duplicate other tests?
2. **Quality**: Uses web-first assertions? Has fragile selectors? Uses hard waits?
3. **Reliability**: Would this test be flaky in CI? Race conditions present?

Create TEST_AUDIT.md with:
- Summary statistics (total files, keep/enhance/remove counts)
- Table of all files with columns: File | Decision | Rationale | Priority Notes
- Decisions: KEEP (good as-is), ENHANCE (has value but needs fixes), REMOVE (duplicate/flaky/low-value)
- Section cataloguing anti-patterns found (with file examples):
  - Hard waits (waitForTimeout)
  - Manual visibility checks (isVisible() without assertion)
  - Fragile selectors (CSS classes without role/label)
  - Missing worker lifecycle management
  - Download race conditions
- Section identifying good patterns to preserve

Look specifically for:
- Duplicate test coverage (e.g., file-conversion-e2e.spec.ts vs file-conversion-e2e-fixed.spec.ts)
- Tests that only console.log results without assertions
- Tests using page.evaluate for actions that have Playwright APIs
- Tests with excessive timeouts (>5000ms hard waits)
